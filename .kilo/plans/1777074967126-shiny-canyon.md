# Plan de Desarrollo: Web Scraper de Competidores (Cloudflare/Local + OpenRouter)

## 1. Arquitectura General
El sistema será una aplicación de consola en Python diseñada para ejecutarse de forma asíncrona, maximizando la velocidad al procesar múltiples URLs en paralelo. 
Tendrá una arquitectura modular que permita intercambiar el motor de scraping entre una solución gestionada (Cloudflare) y una ejecución totalmente local.

*   **Lenguaje:** Python 3.10+
*   **Concurrencia:** `asyncio` para controlar el flujo de tareas concurrentes.
*   **Motores de Scraping Intercambiables:**
    1.  **Cloudflare Worker (Remoto):** Usa `aiohttp` para llamar a un endpoint Serverless (`/crawl`) que corre un navegador headless. Ideal para evitar bloqueos por IP de e-commerces.
    2.  **Playwright (Local):** Motor que lanza navegadores Chromium (headless) de manera asíncrona localmente. Ideal para desarrollo, tests y casos donde no se disponga de Cloudflare o la IP local no esté bloqueada.
*   **Extracción de Datos:** Uso de LLMs a través de la API de OpenRouter (librería `openai` de Python configurada con el base URL de OpenRouter) usando "Structured Outputs" (JSON) para extraer sistemáticamente: `nombre_pagina`, `link`, `nombre_producto` y `precio`.
*   **Manejo de Datos y Output:** `pandas` para exportar a `.csv` y `.xlsx` de manera robusta.

## 2. Fases de Implementación

### Fase 1: Configuración del Proyecto y Dependencias
*   Inicializar un entorno virtual.
*   Crear archivo `requirements.txt` con: `aiohttp`, `playwright` (scraper local), `openai`, `pandas`, `python-dotenv`, `openpyxl`, `beautifulsoup4` (para limpieza de HTML).
*   Instalar navegadores locales con `playwright install chromium`.
*   Configurar archivo `.env` para almacenar de forma segura las credenciales y configuración:
    *   `SCRAPER_ENGINE=cloudflare` (Opciones: `cloudflare` o `local`)
    *   `CLOUDFLARE_WORKER_URL`
    *   `CLOUDFLARE_API_TOKEN`
    *   `OPENROUTER_API_KEY`
    *   `LLM_MODEL=stepfun/step-3.5-flash`

### Fase 2: Configuración del Worker en Cloudflare (Opcional si usa Local)
*   Creación de un script `worker.js` (basado en Puppeteer para Cloudflare Browser Rendering).
*   Configuración de `wrangler.toml` para habilitar el binding de Browser Rendering en tu cuenta de Cloudflare.
*   Pasos documentados para desplegar (`npx wrangler deploy`) y obtener la URL del endpoint `/crawl`.

### Fase 3: Módulos de Scraping (Core)
*   Crear una interfaz/clase base `BaseScraper`.
*   **CloudflareScraper:** Usa `aiohttp` para invocar el Worker de Cloudflare, implementando reintentos (retries) y timeouts en caso de respuestas lentas.
*   **LocalPlaywrightScraper:** Inicia una instancia asíncrona de `playwright`, maneja el pool de contextos de navegador y visita la URL para extraer el HTML renderizado con JS.
*   Una función factoría `get_scraper()` que decida cuál motor instanciar según `SCRAPER_ENGINE` de `.env`.

### Fase 4: Motor de Extracción con LLM (OpenRouter)
*   Crear un módulo `extractor.py`.
*   Limpieza preliminar del HTML devuelto por cualquier scraper (usando `BeautifulSoup` para eliminar `<script>`, `<style>`, `<nav>`, `<footer>`) para reducir radicalmente los tokens enviados al LLM, ahorrando costo y tiempo.
*   Crear el prompt instruyendo al modelo a devolver un objeto JSON estricto (nombre de la página, link, nombre del producto, precio).
*   Función asíncrona `extract_product_info(html_content, url, session)` que llama a la API de OpenRouter, inyectando la variable de entorno `LLM_MODEL`.

### Fase 5: Orquestador Concurrente y Exportación
*   Crear el módulo principal `main.py`.
*   Leer una lista de URLs desde un archivo de texto (`urls.txt`).
*   Usar `asyncio.Semaphore()` (ej. 10 tareas concurrentes) para no exceder los límites de rate-limit de Cloudflare, OpenRouter, o la capacidad de RAM local de Playwright.
*   Tomar los diccionarios con la data procesada y usar `pandas` para guardar los resultados finales en `resultados_competidores.csv` y `resultados_competidores.xlsx`.

## 3. Consideraciones de Optimización
*   **Limpieza de HTML:** Enviar el DOM completo a un LLM es lento y costoso. Se extraerá solo el `<body>` y se filtrará el ruido antes del LLM. Alternativamente, extraerlo como texto plano estructurado (Markdown) si el Worker de Cloudflare lo soporta.
*   **Elección de Modelo:** Se recomendará usar un modelo rápido y económico en OpenRouter (como `gpt-4o-mini`, `claude-3-haiku` o `llama-3-8b-instruct`) ya que la tarea de extracción es sencilla.
*   **Control de Concurrencia:** Lotes paralelos de 5-10 peticiones a la vez para balancear velocidad vs límites de rate-limit de las APIs.
