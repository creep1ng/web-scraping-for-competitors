# Plan de Implementación: Sistema de Logging para Scraper

## 1. Objetivo
Diseñar e implementar un sistema de logging centralizado para el scraper en Python, reemplazando los actuales `print()`. El sistema deberá gestionar múltiples niveles de severidad (INFO, WARNING, ERROR, DEBUG) asegurando que información valiosa como la versión/commit, la URL procesada y el contenido HTML (cuando aplique) queden adecuadamente registrados.

## 2. Requerimientos Cumplidos
*   **Global:** Cada registro de log contendrá el hash del commit actual (o versión) del código desde el cual se ejecuta el scraper.
*   **Nivel DEBUG:** Contendrá información exhaustiva incluyendo la URL, el código HTML descargado, las trazas de error de Python (stack traces) y datos de red o selectores.
*   **Niveles INFO, WARNING, ERROR:** Registrarán eventos generales, incluyendo la URL en proceso, advertencias sobre páginas no escaneadas (ej. fallos en paginación) y los errores críticos de ejecución (ej. fallos de I/O, sin trazas extensas).

## 3. Arquitectura Propuesta

### 3.1. Módulo Central (`logger.py`)
Crearemos un archivo `logger.py` encargado de inicializar la configuración de la librería estándar `logging` de Python.
*   **Obtención del Commit:** Mediante el módulo `subprocess` se ejecutará `git rev-parse --short HEAD` (o leyendo un archivo `VERSION`) al inicializar el logger.
*   **Filtros / Custom Formatter:** Se implementará un `Filter` o un `Formatter` personalizado que inyecte de manera automática el parámetro `commit` a cada registro (`LogRecord`).
*   **Archivos de Salida:** 
    *   `scraper_info.log`: Para niveles INFO, WARNING y ERROR.
    *   `scraper_debug.log`: Para el nivel DEBUG (almacenará el HTML y trazas, por lo que rotará más rápido).
*   **Formato de Salida:** Se recomienda usar un formato estructurado (como JSON a través de `python-json-logger`) para que el HTML y las trazas multilínea no rompan la legibilidad del log, aunque también puede usarse un formato de texto enriquecido.

### 3.2. Adaptación del Código Existente
*   **`main.py`**:
    *   Reemplazar los `print(...)` por `logger.info(...)`.
    *   En las capturas de excepciones (bloques `except Exception as e:`), usar `logger.error("Error crítico...", exc_info=False)` o `logger.warning("Página no escaneada...")`.
    *   Registrar el HTML descargado justo después de obtenerlo: `logger.debug("HTML obtenido", extra={'url': url, 'html': html})`.
*   **Scrapers (`scrapers/`)**:
    *   Registrar errores de red o timeouts usando `logger.error` o `logger.warning`.
    *   Si se captura una excepción imprevista, lanzar `logger.debug("Traza de error", exc_info=True)` para guardar la traza completa.
*   **Extractores (`extractors/`)**:
    *   Reportar como WARNING si una estructura HTML no coincide con el selector esperado.

## 4. Fases de Implementación (Plan de Acción)
1.  **Crear `logger.py`**: Configurar formatos, handlers (consola y archivo rotativo) y la inyección del commit.
2.  **Integrar Logger en `main.py`**: Eliminar los `print()` e inyectar el nuevo logger. Aplicar `extra={'url': url}` en los logs correspondientes.
3.  **Añadir Logs de HTML y Stack Traces (DEBUG)**: Insertar el guardado de `html` y uso de `exc_info=True` en los bloques try/except exclusivamente bajo llamadas `.debug()`.
4.  **Actualizar Dependencias**: Si se decide por logs estructurados en JSON, añadir `python-json-logger` a los requerimientos.
5.  **Revisión y Pruebas**: Ejecutar un scraping de prueba y verificar que el archivo `scraper_debug.log` contiene el HTML y el commit, mientras que la consola muestra solo la actividad regular.
