# Plan: Comparador de Precios con Fuzzy Matching de Productos

## Contexto

- **Productos propios**: ~435 productos exportados desde WooCommerce (`/home/creep/Downloads/wc-product-export-25-4-2026-1777140983496.csv`)
  - Columnas: `Nombre`, `Precio normal`
  - Nombres en MAYÚSCULAS, ej: `"BATERÍA DE GEL GREENPOINT 12V 100AH (agotado)"`

- **Productos competidores**: 923 productos extraídos de 3 páginas (`resultados_competidores.csv`)
  - `emergente.com.co`: 310 productos
  - `ingesolar.com.co`: 491 productos
  - `ineldec.com`: 119 productos
  - Columnas: `nombre_pagina`, `link`, `nombre_producto`, `precio`
  - Nombres en formato título, ej: `"Batería de Gel 12V 100Ah GreenPoint"`

## Problema

Los nombres de productos son similares pero presentan variaciones que impiden un matching exacto:
- Diferente capitalización (MAYÚSCULAS vs Título)
- Diferente orden de palabras (marca al inicio vs al final vs en medio)
- Variaciones en acentos (tildes presentes/ausentes)
- Palabras "ruido" diferentes (`(agotado)`, `SIN GARANTIA`, `– RETIE`, etc.)
- Diferentes preposiciones/artículos (`de`, `para`, `el`, `la`)
- Precios en formatos inconsistentes (`354000`, `$ 1.486.080`, `$ 5.400`)

## Investigación: Funciones de Similitud Evaluadas

### 1. `rapidfuzz` (Recomendada)
- **Biblioteca**: `rapidfuzz` (MIT license, C++ optimizado, ~40% más rápido que fuzzywuzzy)
- **Scorers relevantes**:
  - `fuzz.ratio`: Distancia de Levenshtein pura. Buena para strings cortos/códigos.
  - `fuzz.token_sort_ratio`: Tokeniza, ordena alfabéticamente, luego compara. **Ideal cuando el orden de las palabras varía** (ej: `"GreenPoint Batería 12V"` vs `"Batería 12V GreenPoint"`).
  - `fuzz.token_set_ratio`: Tokeniza, elimina duplicados, compara intersección + diferencias. **Más permisivo, útil cuando hay palabras adicionales** en un nombre pero el core es igual.
  - `fuzz.partial_ratio`: Busca la mejor subsecuencia común. Útil para matches parciales.
- **APIs de utilidad**:
  - `process.extractOne(query, choices, scorer=...)`: Encuentra el mejor match de una lista.
  - `process.cdist(queries, choices, scorer=...)`: Matriz de similitud entre dos colecciones (muy eficiente).
- **Ventaja principal**: Es el estándar actual para fuzzy matching en Python, compatible con fuzzywuzzy pero más rápido y con mejor licencia.

### 2. `thefuzz` / `fuzzywuzzy`
- API casi idéntica a rapidfuzz.
- Desventaja: Licencia GPL, rendimiento inferior.
- **Veredicto**: No usar, reemplazar directamente por `rapidfuzz`.

### 3. `jellyfish`
- Ofrece métricas como **Jaro-Winkler** (bueno para errores tipográficos en nombres cortos) y **Soundex** (similitud fonética).
- Desventaja: No maneja bien diferencias en el orden de palabras ni palabras adicionales.
- **Veredicto**: Complementario, no principal.

### 4. TF-IDF + Cosine Similarity (`scikit-learn`)
- Convierte textos a vectores y mide similitud del coseno.
- Desventaja: No capta bien la importancia de tokens técnicos específicos (12V, 100AH); tiende a dar peso similar a palabras comunes.
- **Veredicto**: No ideal para productos técnicos con especificaciones clave.

### 5. Embeddings Semánticos (`sentence-transformers`)
- Capta significado semántico.
- Desventaja: Overkill para este caso; los productos se diferencian más por specs técnicas (12V vs 24V) que por significado abstracto. Además, requiere dependencias pesadas.
- **Veredicto**: No recomendado para este caso.

## Estrategia de Matching Propuesta (Híbrida)

Dado que los productos son equipos técnicos (baterías, paneles, inversores), **es crítico que no se emparejen productos con especificaciones técnicas diferentes** (ej: una batería 12V 100Ah con una 12V 200Ah del mismo fabricante). La estrategia propuesta es:

### Paso 1: Normalización de Texto
- Minúsculas, eliminar acentos (`unidecode`).
- Eliminar solo palabras de ruido específicas del dominio que aparecen en un dataset pero no en el otro:
  - De productos propios: `(agotado)`
  - De competidores (`ingesolar.com.co`): `sin garantia`, `retie`, `suntree`, `epever`, `leader`, `generico`, `allyce`
- **No usar NLTK stopwords**: Las stopwords genéricas de NLTK (`de`, `la`, `el`, `y`) no son ruido en este caso porque aparecen en ambos lados. Además, `rapidfuzz.token_set_ratio` ya descarta palabras duplicadas y maneja bien las comunes sin necesidad de pre-procesamiento con NLTK.
- Normalizar espacios múltiples.

### Paso 2: Extracción de Features Técnicas (Regex)
Para cada producto, extraer tokens clave que actúen como "filtros duros":
- **Voltaje**: `\d+\s*v` → `12v`, `24v`, `48v`, `220v`
- **Capacidad**: `\d+\s*ah` → `100ah`, `200ah`
- **Potencia**: `\d+\s*(w|kw)` → `100w`, `5kw`
- **Marca**: Diccionario de marcas conocidas (`greenpoint`, `newmax`, `must`, `growatt`, `deye`, `solis`, `luxen`, `trinasolar`, `jinko`, `ja solar`, `apsystems`, `epever`, `srne`, `hoymiles`)
- **Tipo**: `gel`, `litio`, `dzf`, `monocristalino`, `bifacial`, `mppt`, `on-grid`, `hibrido`

### Paso 3: Filtrado de Candidatos
Para cada producto propio, filtrar la lista de competidores a solo aquellos que compartan **al menos 2 features técnicas** (ej: misma marca + misma capacidad, o mismo voltaje + mismo tipo). Esto reduce drásticamente falsos positivos.

### Paso 4: Fuzzy Matching con rapidfuzz
Sobre los candidatos filtrados, aplicar:
- **Scorer primario**: `fuzz.token_set_ratio` (maneja orden diferente y palabras adicionales).
- **Scorer secundario (tie-breaker)**: `fuzz.ratio` cuando hay empates.
- **API**: `rapidfuzz.process.extractOne` para eficiencia.

### Paso 5: Umbral de Confianza
- Aplicar umbral de similitud: **≥ 75** (confirmado por el usuario).
- Matches por debajo del umbral se marcan como "sin match" o "revisión manual".

### Paso 6: Normalización de Precios
- Eliminar `$`, puntos de miles (`.`), espacios.
- Convertir a `int` para comparación aritmética.
